PART 5: ISSUE TEMPLATE TEXT
================================================================================

Use this when creating your GitHub issue:

TITLE:
[Intermittent] Frontend cannot connect to API on dev server restart

BODY:

### Description
Frontend fails to connect to the API (http://localhost:8000) when restarting 
dev servers (`task dev`). The issue occurs intermittently—sometimes the 
connection establishes immediately, other times requires manual intervention 
(browser refresh, server restart).

### Expected Behavior
After running `task dev`, both the FastAPI backend (port 8000) and Vite 
frontend (port 5173) should start cleanly. The frontend should proxy all 
`/api/*` requests to `http://localhost:8000` via Vite's proxy configuration, 
and API calls should succeed immediately.

### Actual Behavior
- Backend starts but frontend cannot reach API endpoints
- Network tab shows failed requests to `/api/*` with connection refused or timeout
- Manual page refresh sometimes resolves it
- Restarting both servers (using `task kill` then `task dev`) sometimes helps
- No clear error messages in browser console or server logs

### Reproduction Steps
1. Start fresh dev environment: `task setup`
2. Run: `task dev`
3. Navigate to http://localhost:5173
4. Attempt to load game list or interact with API endpoints
5. Observe: Intermittent failures, no consistent pattern

### Environment Information
- OS: [Linux/macOS/Windows WSL]
- Node version: [output of `node --version`]
- Python version: 3.11+
- Browser: [Chrome/Firefox/Safari]
- Task command used: `task dev`

### Network Tab Observation
Request URL: http://localhost:5173/api/games
Status: [CORS error / Connection refused / Timeout]
Response Headers: [paste relevant headers or "empty"]

### Diagnostic Script Output
[Run `bash scripts/diagnose_dev_connection.sh` and paste output here]

### What I've Already Tried
- [ ] Ran `task kill` and restarted `task dev`

PART 6: ROOT CAUSE ANALYSIS
================================================================================

MOST LIKELY CAUSE: Vite Proxy Initialization Timing

SYMPTOMS:
- First request after `task dev` finishes fails
- Reloading page after a few seconds succeeds
- Consistently fails on first attempt

WHY THIS HAPPENS:
1. task dev starts both servers in parallel
2. Vite frontend initializes its proxy configuration
3. First browser request immediately attempts API call
4. FastAPI backend may not be fully bound to port 8000 yet
5. Request fails with connection refused
6. Developer refreshes page
7. Backend is ready by now
8. Request succeeds

FIX OPTION 1: Add Client-Side Retry Logic (Recommended)
   File: frontend/src/lib/api.js
   
   Add this function:
   ```javascript
   const fetchWithRetry = async (path, options = {}, retryCount = 0) => {
     try {
       const res = await fetch(path, options)
       if (!res.ok) throw new Error(`HTTP ${res.status}`)
       return await res.json()
     } catch (error) {
       if (retryCount < 3) {
         const delay = 200 * Math.pow(2, retryCount)
         await new Promise(r => setTimeout(r, delay))
         return fetchWithRetry(path, options, retryCount + 1)
       }
       throw error
     }
   }
   ```

FIX OPTION 2: Stagger Server Startup
   File: Taskfile.yml
   
   Add this task:
   ```yaml
   dev:
     desc: Backend and frontend with staggered startup
     cmds:
       - task: dev:backend &
       - sleep 3
       - task: dev:frontend
   ```

FIX OPTION 3: Do Both
   Most robust solution

IMPLEMENTATION TIME: 15-20 minutes

PART 7: DEBUGGING WORKFLOW
================================================================================

When experiencing the issue, follow this sequence:

STEP 1: Run Diagnostic Script (30 seconds)
   bash scripts/diagnose_dev_connection.sh
   
   This checks:
   - Port 8000 (FastAPI) is in use
   - Port 5173 (Vite) is in use
   - Backend /health endpoint responds
   - Frontend server responds
   - API proxy through frontend works
   - localhost DNS resolution
   - Vite proxy config

STEP 2: Check Port Availability (10 seconds)
   lsof -i :8000
   lsof -i :5173
   
   Should show process info, not be empty

STEP 3: Force Kill and Restart (15 seconds)
   task kill
   task dev

STEP 4: Test with curl (10 seconds)
   curl http://localhost:8000/health
   curl http://localhost:5173/api/health
   
   Both should return {"status":"ok"}

STEP 5: Check Browser Network Tab (20 seconds)
   - Open DevTools (F12)
   - Go to Network tab
   - Filter by XHR/Fetch
   - Reload page
   - Look for failed requests
   - Check status code and response

STEP 6: If Still Failing, Create GitHub Issue
   - Use template from PART 5
   - Include diagnostic output
   - Attach Network tab screenshot
   - Document exact reproduction steps

TOTAL TIME: 5 minutes to identify issue
