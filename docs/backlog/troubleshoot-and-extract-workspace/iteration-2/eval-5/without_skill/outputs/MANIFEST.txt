PROBLEM STATEMENT
================================================================================

Batch PDF extraction script hangs around game 15-20 when processing 50+ games.
Error: "connection pool exceeded"
Impact: Complete blocking of batch game enrichment pipeline
Root Cause: Single synchronous Supabase client + unbounded thread concurrency

By game 15, all 10 connection pool slots are occupied, causing remaining
threads to queue indefinitely. Timeouts trigger after 120+ seconds, script
never completes naturally.

================================================================================
SOLUTION SUMMARY
================================================================================

4 code changes, 11 minutes total:

1. CRITICAL (5 min): Add semaphore(3) to limit concurrent DB operations
   File: app/core/supabase.py
   Effect: Prevents thread pool explosion, fixes 80% of issue

2. HIGH (3 min): Add asyncio.timeout(30) to all queries
   File: app/core/supabase.py
   Effect: Forces errors to surface, prevents indefinite hangs

3. MEDIUM (1 min): Increase batch delay from 0.5s to 2.0s
   File: Batch processing script
   Effect: Allows thread pool + connections to cool between batches

4. MEDIUM (2 min): Remove retry backoff, move to Taskfile
   File: batch_processor.py + Taskfile.yml
   Effect: Follows CDD principles, errors surface immediately

RESULT: 50+ games process in 115 seconds (vs. timeout at game 15)

================================================================================
ROOT CAUSES IDENTIFIED
================================================================================

1. CRITICAL: Single sync client in async context (app/core/supabase.py:7)
   - One _client shared via anyio.to_thread
   - Each call spawns thread requesting connection
   - Effect: 20 concurrent requests, 10 connection slots = exhaustion

2. CRITICAL: Unbounded thread concurrency (batch_processor.py:46-48)
   - asyncio.gather(*tasks) with no semaphore
   - 50 items = unbounded thread creation
   - Effect: All threads block waiting for connections

3. HIGH: No timeout on queries (all supabase.py execute() calls)
   - Default 60s, no timeout parameter
   - Slow Supabase = thread holds connection indefinitely
   - Effect: Hangs compound, pool starves further

4. MEDIUM: Exponential backoff in app code (batch_processor.py:118)
   - Retry logic should be in infra, not app
   - 2^attempt = 2s + 4s + 8s sleeps
   - Effect: Errors masked, root cause hidden

5. LOW: Missing connection pool warmup
   - Cold start adds 1-2s to first batch
   - Effect: Minor latency, not critical
