This package contains a complete diagnosis and solution for the Supabase

================================================================================
WHAT HAPPENED
================================================================================

The batch script would hang after processing 15-20 games with the error:
"connection pool exceeded"

Root cause: One synchronous Supabase client shared across unbounded concurrent
threads. By game 15, all 10 connection pool slots were occupied, causing 
remaining threads to queue indefinitely.

================================================================================
THE SOLUTION
================================================================================

4 code changes (11 minutes total):

1. Add semaphore to limit concurrent DB operations (CRITICAL - 5 min)
2. Add timeout to prevent indefinite hangs (HIGH - 3 min)
3. Increase batch delay from 0.5s to 2.0s (MEDIUM - 1 min)
4. Remove retry backoff, move to Taskfile (MEDIUM - 2 min)

Result: Process 50+ games in 115 seconds (vs. timeout at game 15)

KEY INSIGHT
================================================================================

The semaphore is the cure. Without it, every concurrent database operation
spawns a thread requesting a connection from the pool. With 10 connection
slots and 20+ concurrent requests, threads queue indefinitely.

The semaphore limits concurrent DB operations to 3, keeping threads in an
async queue (non-blocking) instead of a thread pool queue (blocking).

This single change (Fix #1) solves 80% of the problem.
